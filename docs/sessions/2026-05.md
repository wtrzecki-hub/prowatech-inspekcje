**Plik tytułowy PDF: pdf-lib + jsPDF — patterny do zapamiętania:**

- Page numbering po pdf-lib insertPage nie aktualizuje się — stopki jsPDF zostają z oryginalnym `Y` (`getNumberOfPages()` w momencie apply footers). Decyzja: live with it. Strony scan PDF i tak nie mają stopki (są wszywane po footer apply), więc niespójność jest wizualnie zaakceptowalna.

**Lekcje sesji (do PROGRESS.md gotchas):**
1. **Multiple uwag w jednym pliku → bundle do jednego PR** — gdy zmiany dotyczą tych samych plików (renderery), próba rozdzielenia per-uwaga generuje konflikty 1:1 line. Lepiej zbundlować i merge'ować pierwsze. Sprawdzone: PR #13 zawierał wszystkie zmiany rendererów (1206 insertions) + 6 fix-ów wizualnych, PR #14-#17 to czyste UI+migracje bez ani jednego konfliktu.

7. **Denormalized fields niesynchronizowane** — `turbines.last_inspection_date/protocol/next_inspection_date` rozjeżdżają się z aktualnym stanem `inspections` (np. **Lekcje sesji (do PROGRESS.md gotchas):**
1. **`gh pr merge` fail w worktree** — `gh pr merge` próbuje checkout brancha lokalnie, fail na "fatal: 'main' is already used by worktree at 'C:/prowatech-inspekcje'". Workaround: `gh api -X PUT repos/{owner}/{repo}/pulls/{N}/merge -f merge_method=squash` + `gh api -X DELETE repos/{owner}/{repo}/git/refs/heads/{branch}`.
2. **`gh api` z leading slash** — `/repos/...` powoduje `invalid API endpoint: "C:/Program Files/Git/repos/..."` (Git Bash on Windows interpretuje `/` jako path prefix). Zawsze bez slash: `repos/...`.
3. **`git checkout {branch} -- file` traci uncommitted edits** — uncommitted changes na bieżącym branchu giną gdy `checkout` z innego brancha pobiera plik. Komituj lokalnie zanim przełączasz.
4. **`addSubHeading` w jsPDF nie zawija** — long titles overflow. Workaround: split na bold + italic body, albo użyj `splitTextToSize` przed `pdf.text`.
5. **Vercel deploy timing** — po `gh api PUT merge`, deploy zaczyna się ~30s później, build ~1-2 min. User może odświeżać i nie widzieć zmian — sprawdzić `gh api repos/.../deployments?per_page=1` (state: success) zanim się odpisze.
6. **Cherry-pick conflict gdy main ma squashed merge** — moje commity z brancha A miały content już zawarty w squashed commit X na main. Cherry-pick mojego nowego commitu Y na fresh branch z main → konflikt bo rebase wymaga 1:1 line context. Rozwiązanie: `git reset --hard origin/main` + `git cherry-pick {nowy-commit}` (bez wcześniejszych commitów z A).
